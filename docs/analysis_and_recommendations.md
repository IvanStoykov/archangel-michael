# Archangel Michael: System Analysis & Vacation Auto-Pilot Blueprint

## 1. The Opus vs. Qwen Reality Check
**The Outcome:** Qwen3.6-35B found a genuine, localized bug (the `ftyp` brand copying) but failed to identify the actual root cause of the system failure (the `RTSPClient.Init` config wipe). Opus correctly graded this as "Right bug, wrong rank, missed the real one."

**The "Why":** Qwen was operating on a **static context bundle**. When you feed a local model a statically bundled set of files and say "find the bug", it assumes the code path it is reading is actually executing. It lacks the dynamic capability to trace the entry point from config down to execution. 

**How to automate and improve this:**
You cannot solve this by making the local model "smarter"; you solve it by giving it a better workflow.
1. **Dynamic Tracing over Static Bundles:** Instead of statically bundling files, the local model needs to use tools (like `grep_search` and `read_file`) iteratively. 
2. **Two-Pass Architecture:**
   * **Pass 1 (The Mapper):** Ask a model to trace the flow of a variable (e.g., "Trace where `videoReturnType` is instantiated and read").
   * **Pass 2 (The Auditor):** Pass the exact files identified in Pass 1 to the deep-dive code auditor.

## 2. Pushing the RTX 4070 (12GB) & 32GB RAM Further
You achieved an incredible 10x speedup by switching to a 35B-A3B MoE with upstream `draft-mtp`. Here is how to squeeze the last 10-15% out of your system:

* **KV Cache Quantization (`q4_0` or `q4_1`):** You are currently using `-ctk q8_0 -ctv q8_0`. Upstream `llama.cpp` now supports `q4_0` and `q4_1` KV caching. At 128K context, moving from Q8 to Q4 KV cache will save *gigabytes* of VRAM, allowing you to increase the `-fitt` headroom or offload more main model layers to the GPU.
* **CPU Thread Tuning (`-t`):** You have `-t 10 -tb 12`. On an i7-13700KF, you have 8 P-Cores. Setting `-t` higher than your physical P-cores forces context switching, which actually *slows down* CPU offloading. Change this to `-t 8 -tb 8` (bind strictly to P-cores).
* **Numa / CPU Pinning:** If you use `--no-mmap`, ensure `llama-server` is bound to the P-cores.
* **Batch Sizes:** Your `-b 1024 -ub 512` is solid, but if you switch to Q4 KV cache, try `-ub 1024` to see if the larger micro-batch speeds up prompt ingestion without OOMing.

## 3. DX, Orchestration, and Logging (Moving away from Bash)
Bash scripts (`run_audits.sh`, `check_status.sh`) are fragile, lack proper concurrency control, and are hard to debug when a task hangs at 3 AM. 

**The Recommended Go Architecture:**
Since you like Golang, we should replace the bash orchestration with a single Go daemon (`archangel-server`).
* **Process Management:** Use Go's `os/exec` to manage the `llama-server` lifecycle. Go can automatically detect if the server crashes and restart it.
* **Task Queue:** Use a lightweight embedded database (SQLite via `crawshaw/sqlite` or `mattn/go-sqlite3`) to track task status (`pending`, `running`, `completed`, `failed`). This replaces the `.run_audits.lock` and `.jsonl` files and allows for robust retries.
* **Structured Logging:** Use Go's `log/slog`. You can stream these logs to a frontend or write them to a structured JSON file.
* **Hermes API:** Go can make direct HTTP calls to the `llama-server` (or manage the Hermes CLI via `os/exec`), parsing the results robustly.

## 4. Integrating Frontier Models (Cursor, Gemini, Claude)
Your local models are your "grunts" (free, unlimited tokens, run overnight). Your frontier models (Cursor, Gemini, Claude) are your "Staff Engineers."

**The Orchestration Pipeline:**
1. **Local LLM (Qwen 35B):** Runs the "VideoBytes Marathon". It reads every function and outputs a markdown file of potential bugs, anomalies, and structural issues.
2. **Gemini CLI / Claude Code:** You write a prompt for Gemini/Claude: *"Read all the `vb_audit_*.md` files generated last night. Filter out the false positives. Identify the 3 most critical architectural flaws across the codebase, and write a Go patch for the highest priority one."*
3. **Cursor CLI:** You use Cursor when you sit back at your desk. You review the diffs generated by Gemini/Claude, accept/reject them, and run your tests.

## 5. Frontend UI (Keeping VRAM Free)
If you want a dashboard to monitor this while at the wedding, **do not build a heavy frontend that competes for system resources.**

**Recommendation: Go + HTMX / Templ**
* **Why:** It runs entirely inside the Go orchestrator binary. It requires **0 MB** of VRAM and negligible RAM. 
* **How:** Your Go orchestrator serves a simple web page. HTMX automatically polls the Go server for SQLite task queue updates. You can view the live status of your overnight marathon from your phone via a simple reverse proxy (like Tailscale or Cloudflare Tunnels).
* **If you strictly want Flutter:** Build a Flutter Web/Desktop app, but compile it in release mode. Have it poll the Go backend's REST API. Keep the Flutter app closed when you don't actively need to look at it to save GPU RAM overhead (since Flutter uses hardware acceleration for UI rendering).

## 6. The "Vacation Auto-Pilot" Blueprint
To have this run autonomously while you are at the wedding:

1. **Weekend Project:** Convert `run_audits.sh` into a simple Go worker pool.
2. **The Queue:** Load `tasks.manifest.yaml` into the Go SQLite DB.
3. **The Execution Loop:**
   * Go starts `llama-server`.
   * Go picks a task, runs Hermes CLI, and saves the `.md` output.
   * If Hermes hangs (timeout), Go kills the process, restarts `llama-server`, and retries.
4. **The Handoff:** Once all local tasks finish, the Go script triggers a `gemini-cli` or `claude` command to ingest the results directory and summarize the highest-value insights into a `vacation_summary.md` file.
5. **Remote Access:** Set up Tailscale on your home PC and your phone. You can access the Go HTMX dashboard from the wedding to watch Qwen churn through thousands of lines of code.
